# Install Git Hooks

## Overview

Install pre-push, pre-commit, and post-commit git hooks with:
- **LLM code review** before push (Qwen via DashScope)
- **Push approval** via Feishu or Telegram interactive message
- **Commit notifications** sent automatically
- **PKGBUILD protection** in `repo` folders

## Quick Start

```bash
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

## ⚠️ Hook Usage Policy

**Default: auto-install hooks on every relevant repo.** Install automatically when:
- User clones from AUR (`https://aur.archlinux.org/*.git`)
- User clones from arch4edu (`https://github.com/arch4edu/*`)
- Working in a `repo/` directory
- Any repo that contains PKGBUILD files

**Only ask the user if:**
- There are **existing non-symlink hooks** that would be overwritten
- The repo is clearly unrelated to AUR/arch4edu work

**Never silently bypass** hooks on the user's behalf. If a push is blocked, explain the cause and suggest fixes. The user decides whether to use `--no-verify` or fix the underlying issue.

```bash
# Default install (auto-run for AUR/arch4edu repos)
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

## Usage Scenarios

### Scenario 1: Clone a new AUR package

```bash
git clone https://aur.archlinux.org/some-package.git
cd some-package
```

**Action**: Always install hooks after cloning. This ensures:
- PKGBUILD modifications are blocked if working in a `repo` folder
- Commit diffs are sent for review before push
- Push requires approval (no accidental direct pushes)

```bash
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

### Scenario 2: Clone from arch4edu/cactus or any arch4edu repo

```bash
git clone https://github.com/arch4edu/cactus.git
cd cactus
```

**Action**: Always install hooks. arch4edu repos manage AUR packages — the PKGBUILD protection and push approval are critical to prevent broken builds.

```bash
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

### Scenario 3: Existing repository — check if hooks are configured

```bash
cd /path/to/existing-repo
```

Check current hook status:

```bash
# Check if hooks are symlinks to our install-git-hooks scripts
ls -la .git/hooks/pre-push .git/hooks/pre-commit .git/hooks/post-commit 2>/dev/null
```

Expected output shows symlinks pointing to `~/.qoder/skills/install-git-hooks/scripts/`:

```
pre-push -> /home/user/.qoder/skills/install-git-hooks/scripts/pre-push
pre-commit -> /home/user/.qoder/skills/install-git-hooks/scripts/pre-commit
post-commit -> /home/user/.qoder/skills/install-git-hooks/scripts/post-commit
```

If hooks are missing or not symlinks, reinstall:

```bash
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

### Scenario 4: Batch check all git repos

Scan common directories for repos missing hooks:

```bash
find ~/aur ~/projects -maxdepth 2 -name .git -type d 2>/dev/null | while read gitdir; do
    repo=$(dirname "$gitdir")
    hook="$gitdir/hooks/pre-push"
    if [ ! -L "$hook" ]; then
        echo "❌ Missing hooks: $repo"
    else
        echo "✅ OK: $repo"
    fi
done
```

### Rule of thumb

**Any git repo that touches PKGBUILD or AUR packages should have these hooks installed.** This includes:
- `git clone` from AUR (`https://aur.archlinux.org/*.git`)
- `git clone` from arch4edu (`https://github.com/arch4edu/*`)
- Local `repo/` working directories
- Any fork or personal package management repo

## First-Time Setup

When `config.py` is missing (gitignored), the hooks will fail to import. Run the setup interactively:

### Step 1: Check if config exists

```bash
ls ~/.qoder/skills/install-git-hooks/scripts/config.py 2>/dev/null || echo "NOT FOUND — run setup"
```

### Step 2: Interactive setup

If config is missing, guide the user through these questions:

1. **Notification backend**: `feishu` or `telegram`?
2. **Backend credentials**:
   - Feishu: `app_id`, `app_secret`, `dest` (open_id or chat_id)
   - Telegram: `bot_token`, `chat_id`, optional `allow_from` (user ID whitelist)
3. **LLM API**: DashScope API key (or use default)
4. **Redis**: host/port (default localhost, or skip)

Then generate `config.py`:

```python
# Generated by install-git-hooks setup
notification_backend = "feishu"  # or "telegram"

# Feishu config
feishu_app_id = "cli_xxx"
feishu_app_secret = "xxx"
feishu_dest = "ou_xxx"

# Telegram config
# telegram_bot_token = "123456:ABC-xxx"
# telegram_chat_id = "123456789"
# telegram_allow_from = [123456789]

# LLM config
llm_model = "qwen3.7-max"
llm_api_url = "https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions"
llm_api_key = "sk-xxx"

# Redis (optional)
redis_host = "127.0.0.1"
redis_port = 6379
redis_db = 0
redis_password = None
```

Save to `~/.qoder/skills/install-git-hooks/scripts/config.py`.

### Step 3: Install hooks

```bash
cd /path/to/repo
bash ~/.qoder/skills/install-git-hooks/install-git-hooks
```

## Debugging

Test hooks in an isolated temp repo without affecting any real repository:

```bash
# Create a temp repo with no remote
TMPDIR=$(mktemp -d)
cd "$TMPDIR"
git init
git commit --allow-empty -m "test"

# Install hooks
bash ~/.qoder/skills/install-git-hooks/install-git-hooks

# Test pre-commit
touch testfile && git add testfile
git commit -m "test commit"

# Test post-commit (should fire automatically)

# Test pre-push (will fail gracefully without remote)
# Or create a local bare repo to simulate:
git init --bare /tmp/test-remote.git
git remote add origin /tmp/test-remote.git
git push  # This triggers pre-push

# Cleanup
rm -rf "$TMPDIR" /tmp/test-remote.git
```

### Common issues

| Symptom | Cause | Fix |
|---------|-------|-----|
| `ModuleNotFoundError: No module named 'config'` | config.py missing | Run interactive setup |
| `Failed to get token` | Wrong Feishu credentials | Check app_id/app_secret |
| Hook not firing | Not a symlink | Run install script, don't copy manually |
| `pip install redis` missing | Redis module not installed | `pip install redis` |

## Switching Backend

Edit `config.py`:

```python
notification_backend = "telegram"  # change from "feishu"
```

### Telegram Setup

1. **Create a bot**: Message [@BotFather](https://t.me/BotFather) on Telegram, send `/newbot`, follow the prompts. Copy the `bot_token`.
2. **Get your chat_id**: Send a message to your bot, then visit:
   ```
   https://api.telegram.org/bot<TOKEN>/getUpdates
   ```
   Find your `chat.id` in the JSON response.
3. **Fill in `config.py`**:
   ```python
   telegram_bot_token = "123456:ABC-DEF..."
   telegram_chat_id = "123456789"
   telegram_allow_from = []  # empty = allow all, or [user_id1, user_id2] to whitelist
   ```

### Telegram vs Feishu Approval Flow

| Feature | Feishu | Telegram |
|---------|--------|----------|
| Approval method | Emoji reactions on message | Inline keyboard buttons (✅ Approve / ❌ Reject) |
| After click | Bot echoes the same emoji back | Message auto-updates to show result, buttons removed |
| Timeout | Bot adds 🚨 reaction | Message edited to show timeout notice |
| Urgent | In-app urgent notification | Message pinned in chat |

Both backends share the same LLM review, PKGBUILD protection, and commit notification logic.
